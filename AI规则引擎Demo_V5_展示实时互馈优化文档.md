# AI 规则引擎 Demo V5：展示与实时互馈优化文档

> 文档定位：这是在现有代码已经完成多个阶段后的**增量优化合同**。  
> 目标不是重写项目，也不是替换 V4 的核心规则引擎，而是在现有基础上修正交互体验、展示结构、实时反馈、How 页解释方式和技能动画映射。

---

## 0. 本轮核心判断

当前 Demo 已经具备基础规则引擎和 Play / How / Why 页面，但测试反馈显示：

1. UI 展示仍像工程原型，作品集展示感不足。
2. Play 页布局没有突出“构筑 → 互馈 → 技能变化 → 动画反馈”的核心体验。
3. 技能种子不能充分重复选择，破坏了“同方向重复固化”的玩法可能。
4. 必须填满 4 个技能位才能生成结果，不符合“每加入一个技能都会改变构筑”的体验目标。
5. How 页固定展示样例，没有跟随 Play 页当前构筑变化，容易让访问者误以为只是静态说明。
6. 技能动画仍偏向旧版单技能/阶段模型，没有充分表达向量结果、技能形态、指向性、范围、段数和互馈变化。

本轮优化的核心方向是：

```text
从“填满 4 个槽后点击生成”
改成
“任意 1–4 个技能种子实时生成；每次新增/替换/重排都会让已有技能与后续技能发生可视化变化”。
```

## 1. 执行方式要求：先审阅代码，再实施优化

由于当前代码结构未知，coding agent 不应直接按本文档臆测文件路径修改。必须先进行一次 code-grounded audit。

### 1.1 Audit 必须回答

coding agent 需要先检查现有代码并输出：

1. 当前 Play 页组件结构。
2. 技能种子池的数据结构。
3. 技能槽位的数据结构。
4. 是否存在“已选后禁用 / 不可重复选择”的逻辑。
5. 是否存在“必须 4 个槽全满才能 generate”的逻辑。
6. generateSkill / generateBuild 的入口函数在哪里。
7. 当前 How 页是否使用固定 mock 数据。
8. 当前动画参数如何从技能结果映射到 Three.js。
9. 当前全局状态是否可被 Play / How 共享。
10. 哪些文件适合修改，哪些文件不应该动。

### 1.2 Audit 后再给出实施计划

coding agent 必须先输出：

```text
- Code Evidence
- Current Behavior Mapping
- Proposed Changed Files
- Risk Assessment
- Step-by-step Implementation Plan
```

得到确认或自行完成审阅后，再进入修改。

## 3. Play 页核心修正

---

### 3.1 技能种子池 UI 升级

当前问题：技能种子池看起来过于简陋，无法表达“元素 + 固化方向 + 行为倾向”。

建议方向：

1. 按元素分组展示，例如：火、冰、雷、岩、影、风。
2. 每个元素下面展示 4 个固化方向。
3. 每个技能种子卡片至少显示：
   - 元素名
   - 固化方向名
   - 1–3 个行为倾向短标签
   - 颜色/符号/纹理提示
4. 卡片不应只是普通按钮，应具有“可铭刻种子”的质感。
5. 种子池应明确传达：种子是可复制的，不是一次性消耗物。

示例表达：

```text
火系
[爆焰] 冲击 / 爆裂
[灼流] 持续 / 蔓延
[炎域] 区域 / 压制
[烬印] 标记 / 延迟
```

注意：不要把行为标签做成最终技能预览。它们只是“种子倾向”，不是生成结果。

---

### 3.2 支持同一个方向重复选择

当前问题：如果选择火系，似乎只能把火的 4 个方向都选上，不能 4 个槽全部放同一个方向。

必须修正：

1. 同一个技能种子可以被放入多个技能位。
2. 同一个元素可以重复。
3. 同一个元素的同一个固化方向也可以重复。
4. 种子池不应因为某个种子被选中而禁用该种子。
5. 拖拽系统如果依赖唯一 id，应区分：
   - seedId：种子类型 id，例如 `fire_burst`
   - instanceId：槽位中的实例 id，例如 `slot_2_fire_burst_169...`

允许案例：

```text
fire_burst → fire_burst → fire_burst → fire_burst
fire_domain → fire_domain → fire_flow → fire_mark
lightning_chain → lightning_chain → lightning_chain → wind_pull
```

验收要求：

- 四个槽可以全部是同一个技能种子。
- 重复构筑仍然可以实时生成结果。
- 对比模式能正确显示重复种子，而不是合并或去重。

---

### 3.3 技能位显示升级

当前问题：放入技能槽位后显示很 low，缺少“技能正在被固化/互馈”的感觉。

建议每个技能位卡片展示：

1. 技能位编号：技能 I / II / III / IV。
2. 当前技能种子：元素 + 固化方向。
3. 当前生成后的技能名称。
4. 主能力 / 次能力 / 变异能力摘要。
5. 被前面技能影响的提示。
6. 被后面技能改写的提示。

示例：

```text
技能 II
灼流之火
生成：灼流爆燃
主能力：持续灼烧
前向塑形：受到 技能 I「爆焰」影响，获得爆裂收束
后向改写：被 技能 III「烬印」追加延迟引爆条件
```

若某个技能位为空，应显示：

```text
拖入技能种子后，该技能会立即参与构筑互馈
```

---

### 3.4 从“点击生成”改为“实时生成”

当前问题：必须放满 4 个技能槽才能生成结果。这个行为不符合当前设计。

必须修正：

1. 只放入第 1 个技能种子时，就应生成一个基础技能结果。
2. 放入第 2 个技能种子后，系统应立即重新计算技能 I 和技能 II。
3. 放入第 3 / 第 4 个技能种子后，同样立即重新计算整个构筑。
4. 拖拽重排、替换、移除某个技能种子时，也应立即重新计算。
5. 原“铭刻生成”按钮可以改为：
   - 保存到对比
   - 重播动画
   - 锁定当前构筑
   - 随机填充

推荐交互：

```text
用户拖入技能种子
  ↓
立即触发 generateBuild(currentSlots)
  ↓
结果面板更新
  ↓
技能位卡片展示变化
  ↓
动画区自动播放或显示“变化已更新，可重播”
  ↓
How 页同步显示当前构筑的计算过程
```

验收要求：

- 不需要点击按钮即可看到结果。
- 1 个技能、2 个技能、3 个技能、4 个技能均可生成。
- 空槽不会导致报错。
- partial build 的 trace 也必须完整可解释。

---

### 3.5 技能变化的可视化反馈

新增/替换/重排技能种子时，不应只是静态刷新文本。应展示“构筑发生了变化”。

建议增加 `previousBuildResult` 与 `currentBuildResult` 的 diff 机制。

需要比较：

1. 技能名称是否变化。
2. 主能力是否变化。
3. 次能力是否新增/消失。
4. 行为向量 Top 维度是否变化。
5. 元素占比是否变化。
6. 动画参数是否变化。

前端可视化方式：

- 新增词条短暂高亮。
- 消失词条淡出。
- 参数条从旧值动画过渡到新值。
- 技能位之间出现短暂箭头脉冲。
- 被后续技能改写的旧技能位出现“回流”提示。

示例文案：

```text
技能 I 被新加入的 技能 III 改写：获得「延迟爆裂」
技能 II 的主能力从「灼烧附着」转为「炎域蔓延」
构筑整体的 zone +0.18，delay +0.12
```

## 4. Play 页布局优化

---

### 4.1 当前布局问题

当前双列布局导致：

1. 左侧构筑设置很短，空间浪费。
2. 右侧技能描述占据大量高度。
3. 技能动画和构筑对比被挤到下方，需要滚动才能看到。
4. 第一屏没有同时展示“选择、结果、动画”三件事。

---

### 4.2 推荐布局方案 A：上方构筑，下方三栏展示

这是最推荐方案。

```text
┌──────────────────────────────────────────────┐
│ 顶部：技能种子池 + 4 个技能位                 │
│ 横向单列，占满页面宽度                         │
└──────────────────────────────────────────────┘

┌───────────────┬───────────────┬───────────────┐
│ 构筑结果       │ 技能动画       │ 互馈/对比       │
│ 技能列表       │ Three.js       │ arrows/compare │
│ 参数变化       │ replay         │ trace summary  │
└───────────────┴───────────────┴───────────────┘
```

优点：

1. 第一屏同时能看到构筑输入、结果、动画。
2. 技能种子池不会被压窄。
3. 动画区能获得足够宽度。
4. 对比区不会被完全挤到页面底部。

---

### 4.3 推荐布局方案 B：左侧构筑，右侧上下分区

如果当前代码更适合保留左侧结构，可以采用：

```text
┌──────────────┬─────────────────────────────┐
│ 左：构筑区    │ 右上：动画展示               │
│ 种子池/槽位   ├─────────────────────────────┤
│              │ 右下：技能结果 + 对比 + How摘要│
└──────────────┴─────────────────────────────┘
```

但需要避免左侧过短。左侧可以包含：

- 种子池
- 技能位
- 随机/清空/保存对比
- 当前构筑指纹摘要

---

### 4.4 响应式要求

桌面端优先作品集展示：

- 大屏可使用三栏。
- 中屏可使用两栏。
- 小屏应折叠为单列。

需要注意：不要使用容易导致横向溢出的 `100vw` 布局。优先使用 `width: 100%`、`max-width`、`box-sizing: border-box`。

## 5. 技能动画的核心优化：引入动画语义向量

---

### 5.1 当前问题

现有动画似乎仍然更匹配旧版逻辑，即一个技能或固定类别动画，而不是当前“技能位互馈 + 向量解码”的逻辑。

如果只用标签决定动画，例如：

```text
有 burst → 播放爆炸
有 chain → 播放闪电链
有 zone → 播放地面圈
```

会导致动画和技能细节脱节。

---

### 5.2 建议新增 AnimationVector

不要把动画只看成标签映射，而应从最终技能语义状态中生成一个 `AnimationVector`。

推荐维度：

```ts
AnimationVector = {
  shapeCircle,       // 圆形范围
  shapeCone,         // 扇形
  shapeLine,         // 直线 / 贯穿
  shapePoint,        // 单点
  shapeChain,        // 连锁路径
  range,             // 释放距离
  radius,            // 影响半径
  segmentCount,      // 段数 / 多段命中
  projectileCount,   // 弹体数量
  travelSpeed,       // 运动速度
  duration,          // 持续时间
  delay,             // 延迟触发
  persistence,       // 地面/空间残留
  targetLock,        // 指向性 / 锁定
  knockback,         // 击退感
  impactIntensity    // 命中强度
}
```

这些不是最终数值平衡，而是视觉表现参数。

---

### 5.3 行为向量到动画向量的映射思路

示例映射：

```text
impact / burst 高
→ 命中强度更高，爆裂更明显，duration 更短

spread / zone 高
→ radius 更大，persistence 更高，shapeCircle 或 shapeCone 增强

pierce 高
→ shapeLine 增强，range 增加，路径更直

chain 高
→ shapeChain 增强，segmentCount 增加，出现折线路径

delay / mark 高
→ 先出现标记，再延迟触发二段效果

pullPush / mobility 高
→ 轨迹移动更明显，可能从圆形变成扇形或冲刺线

control / bind 高
→ 出现环、锁、壁、压制区域
```

重点：

```text
动画形态不应该只由元素决定，而应该由元素 + 行为向量 + 候选能力共同决定。
```

---

### 5.4 示例：圆形近距离伤害如何被互馈改变

基础技能：

```text
火系炎域
→ 圆形近距离范围伤害
```

加入风系风引后：

```text
shapeCircle 降低
shapeCone 增强
range 增加
pullPush 增加
travelSpeed 增加

视觉结果：
圆形火场被风拉成扇形火浪，距离略微增加，并带击退线。
```

加入雷系电涌后：

```text
shapeChain 增强
segmentCount 增加
impactIntensity 增强

视觉结果：
火场中出现跳跃电弧，命中后产生多段爆燃。
```

加入烬印后：

```text
delay 增强
mark 增强
impactIntensity 后置增强

视觉结果：
目标身上先出现烬印，短暂延迟后爆裂。
```

---

### 5.5 多技能 Build 的动画展示策略

由于现在 4 个槽是 4 个独立技能，不建议一次动画只播放一个“总技能”。

推荐两种模式：

#### 模式 A：展示当前选中技能位

- 用户点击技能 I / II / III / IV。
- 动画区播放该技能在当前构筑互馈后的表现。
- 技能位变化后，默认播放变化最大的技能。

#### 模式 B：Build 连续演示

- 依次播放技能 I → 技能 II → 技能 III → 技能 IV。
- 每个技能带有被互馈改写后的效果。
- 适合作品集录屏。

MVP 优先建议：

```text
先实现模式 A，保留一个“播放整套 Build”按钮作为增强项。
```

## 6. How 页优化

---

### 6.1 当前问题

How 页如果只展示固定案例，而不反映 Play 页当前选择，会产生误会：

```text
访问者可能认为 Play 页结果和 How 页解释没有关系，How 页只是静态说明。
```

---

### 6.2 How 页必须接入当前构筑状态

How 页应默认显示 Play 页当前构筑的计算过程。

如果当前没有构筑，则显示默认案例。

推荐逻辑：

```text
if currentBuild has at least 1 seed:
  show current build trace
else:
  show default example trace
```

页面上应明确显示：

```text
当前展示：来自 Play 页的实时构筑
```

或：

```text
当前展示：默认示例构筑
```

---

### 6.3 How 页展示结构建议

How 页应从“文字解释”升级为“可视化计算过程”。

推荐结构：

```text
1. 当前技能位序列
2. 初始语义向量
3. 技能位互馈图
4. 前向塑形结果
5. 后向改写结果
6. 每个技能位最终向量 Top 维度
7. 候选能力竞争表
8. 元素合法性过滤说明
9. 最终能力组
10. 动画向量摘要
```

---

### 6.4 技能之间影响的动画表达

How 页可以加入“互馈箭头”动画：

1. 前向影响：从左到右箭头。
2. 后向改写：从右到左箭头。
3. 箭头粗细表示影响强度。
4. 箭头颜色可以跟来源元素一致。
5. 鼠标悬停箭头时，显示影响摘要。

示例：

```text
技能 I「炎域」 → 技能 II「雷网」
前向塑形：zone +0.18，burst +0.09

技能 III「烬印」 → 技能 I「炎域」
后向改写：delay +0.14，mark +0.11
```

不要求本轮做复杂图算法，可以先用 SVG / CSS arrows / simple div connectors 实现。

---

### 6.5 How 页不应暴露过多工程噪声

How 页目标是让非技术面试官理解：

```text
技能不是查表，而是通过向量语义、技能位互馈、候选能力竞争生成。
```

所以不建议展示大段 JSON。可以折叠到“开发者详情”。

默认展示应是：

- Top 维度
- 影响箭头
- 分数变化
- 最终能力
- 为什么这个能力合法

## 7. Why 页修正

当前问题：Why 页最右侧似乎被页面内滑块挡住，页面占幅和 Play / How 不一致。

coding agent 需要检查：

1. 是否使用了 `100vw` 导致出现垂直滚动条后横向溢出。
2. 是否某个容器有固定宽度超过父容器。
3. 是否没有设置 `box-sizing: border-box`。
4. 是否页面内滚动容器和外层 body 滚动冲突。
5. 是否右侧 padding 不足。

修正原则：

```text
Play / How / Why 三个页面应共用一致的 page shell。
```

建议：

1. 使用统一 `PageContainer`。
2. 设置统一 `max-width`。
3. 设置统一左右 padding。
4. 避免 `width: 100vw`。
5. 对滚动区域使用 `overflow-x: hidden`，但不要掩盖真正布局错误。

验收要求：

- Why 页右侧内容不被滚动条遮挡。
- Play / How / Why 页面宽度观感一致。
- 不出现横向滚动条。

## 8. 对比模式优化

当前对比区被迫下滑看到，展示价值不足。

建议：

1. 对比栏可以作为右侧第三栏或底部 sticky panel。
2. 保存当前构筑后，对比区展示：
   - 构筑序列
   - 每个技能的主能力
   - 构筑指纹
   - 差异最大的向量维度
   - 差异最大的动画参数
3. 对比不应只比较最终技能名，也要比较：
   - 技能互馈路径不同
   - 哪个技能被后向改写
   - 哪个技能受前向塑形更强

示例：

```text
构筑 A：fire_burst → fire_burst → fire_mark → wind_pull
构筑 B：wind_pull → fire_mark → fire_burst → fire_burst

主要差异：
- A 的爆发集中在后段引爆
- B 的火系技能被风系提前塑形，范围更偏扇形推进
- A 的 delay 更高，B 的 range / pullPush 更高
```

## 9. 状态设计建议

现有代码具体结构未知，但建议形成类似状态：

```ts
BuildState = {
  slots: Array<SkillSeedInstance | null>,
  activeSkillIndex: number,
  currentBuildResult: BuildResult | null,
  previousBuildResult: BuildResult | null,
  savedComparisons: BuildResult[]
}
```

每次 slots 变化：

```text
1. previousBuildResult = currentBuildResult
2. currentBuildResult = generateBuild(slots.filter(Boolean))
3. computeBuildDiff(previousBuildResult, currentBuildResult)
4. update Play result
5. update animation params
6. update How trace
```

如果项目使用 Zustand，可以放在 store。  
如果项目只用 React state，也可以提升到 App 层或共享 context。

关键要求：

```text
Play 页和 How 页必须读取同一个 currentBuildResult / trace。
```

9. Why 页不再被右侧滚动条遮挡。

```text
你是一名前端工程师。当前项目是 AI 规则引擎 Demo：技能符文构筑实验室。项目已经完成多个阶段代码，现在需要在不重写项目的前提下，进行展示、实时互馈和动画表现优化。
```

## 13. 本轮优化的判断标准

本轮不是追求更多内容，而是追求访问者在 30 秒内理解：

```text
我拖入一个技能种子，就能看到一个技能。
我再拖入第二个技能，前一个技能也变了。
我可以重复放入同一个方向，形成极端构筑。
不同顺序不是换名字，而是真的改变了能力、参数和动画形态。
How 页能解释刚才发生了什么。
```

这才是该 Demo 最应该展示的核心价值。
