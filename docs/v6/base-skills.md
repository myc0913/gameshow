# V6 基础技能与参数模型

## 1. 设计目标

维度不是越多越好，也不是越少越清楚。V6 将“技能是什么”和“技能有多强”拆成三层：

```text
固定身份层：主元素、主形态、主效果
连续参数层：10 个可增减数值
离散机制层：燃烧、冻结、连锁、推拉等可感知机制
```

这样处理后：

- 雷不需要为了填满维度而拥有推拉。
- 岩不需要天然导电。
- 投射物不会因为数值变化变成领域。
- 同一个技能仍然可以通过参数和机制发生明显变化。
- UI 可以只展示真正发生变化的维度，不需要永远展示一排近似数值。

## 2. 固定身份层

### 2.1 元素

```ts
type ElementKey =
  | "fire"
  | "frost"
  | "lightning"
  | "stone"
  | "shadow"
  | "wind";
```

`element` 是技能的主元素，整个计算过程中保持不变。

互馈可以为当前技能增加一个视觉或机制上的 `accentElement`，但该混合元素不得作为新来源向后传播。

### 2.2 技能形态

```ts
type SkillForm =
  | "projectile"
  | "cone"
  | "zone"
  | "chain"
  | "movement"
  | "construct"
  | "mark"
  | "summon"
  | "line";

type AnimationTrajectory =
  | "none"
  | "straight"
  | "arc"
  | "drop"
  | "ground";
```

形态是只读字段。它决定基础动画模板、空间语义和可接受参数，但不决定元素。

| 形态 | 语义 | 可明显表达的参数 |
|---|---|---|
| `projectile` | 从施法者飞向目标并命中 | 速度、射程、爆炸范围、穿透 |
| `cone` | 从施法者向前持续覆盖扇区 | 范围、持续、覆盖角度、控制 |
| `zone` | 在一块区域中持续生效 | 半径、持续、控制、传播 |
| `chain` | 在多个目标间跳跃 | 跳跃数、传播、速度、衰减 |
| `movement` | 施法者自身移动或突进 | 位移距离、速度、保护、穿透 |
| `construct` | 生成墙、垒或其他实体 | 宽度、持续、保护、阻挡 |
| `mark` | 在目标或位置上留下待触发印记 | 持续、触发强度、控制、延迟爆发 |
| `summon` | 生成一个持续作用的中心体 | 持续、作用范围、控制、力量 |
| `line` | 沿地面或空间直线连续出现 | 长度、速度、穿透、力量 |

### 2.3 基础主效果

每个技能必须有一句不依赖其他符文也能成立的 `coreEffect`。

互馈不能替换这句话的主语义，只能：

- 调整其中的强度、范围、速度、持续等参数。
- 加入至多一个强前向机制。
- 加入一个由后位技能联合得到的轻量修饰包。
- 改变命名后缀、材质、粒子和命中反馈。

## 3. 连续参数层

内部统一使用 `0..1`，数据文档为便于阅读使用 `0..100`。进入引擎时除以 100。

```ts
type StatKey =
  | "power"
  | "reach"
  | "area"
  | "duration"
  | "speed"
  | "control"
  | "force"
  | "propagation"
  | "penetration"
  | "protection";
```

| 参数 | 含义 | 不代表什么 |
|---|---|---|
| `power` | 伤害、爆发或效果总强度 | 不自动代表范围 |
| `reach` | 技能能触及多远 | 不自动代表飞行速度 |
| `area` | 一次覆盖的空间大小 | 不改变技能形态 |
| `duration` | 效果、实体或残留存在时间 | 不自动增加伤害 |
| `speed` | 飞行、展开、移动或触发速度 | 不自动代表连锁 |
| `control` | 限制行动或稳定命中的能力 | 不指定推、拉、冻哪一种 |
| `force` | 冲击、推动、拉扯或打断的物理感 | 只有能力边界允许时才解码成推拉 |
| `propagation` | 向额外目标或空间继续扩散的能力 | 不等于面积 |
| `penetration` | 穿过目标、障碍或防护的能力 | 不等于伤害 |
| `protection` | 护盾、阻挡、减伤或施法安全性 | 不等于控制 |

### 3.1 为什么不继续使用混合 16 维

旧维度同时混入了形态、战斗数值、动画和状态语义，容易出现：

- 数值变了但用户不知道变化是什么意思。
- 为了让每个维度都有值，元素获得不符合直觉的能力。
- 动画根据最高维度变形，导致主形态不稳定。

V6 的 10 个参数只负责可连续变化的量，不负责回答“是什么”。

## 4. 离散机制层

```ts
type MechanicKey =
  | "burn"
  | "chill"
  | "freeze"
  | "shock"
  | "stun"
  | "knockback"
  | "pull"
  | "pierce"
  | "guard"
  | "mark"
  | "delayedBurst"
  | "chain"
  | "fracture"
  | "bind"
  | "obscure"
  | "echo"
  | "haste";
```

机制强度同样使用 `0..1`。机制不是简单标签：

- `0`：不存在。
- `0..0.14`：只进入细节和轻微视觉，不作为卡片主标签。
- `0.15..0.34`：显示为次级特征。
- `0.35..0.69`：显示为明显机制。
- `0.70..1`：可以成为技能的主要差异点。

这些显示分段只用于信息层级，不是计算门槛。计算仍然连续。

## 5. 元素能力边界

每个元素不是平均覆盖所有机制，而是有明确能力范围。

| 元素 | 天然擅长 | 可以通过合适关系获得 | 默认禁止直接产生 |
|---|---|---|---|
| 火 | `power`、`area`、`duration`；燃烧、延迟爆发 | 扩散、蒸汽遮蔽、熔裂 | 冻结、拉拽、护墙 |
| 冰 | `control`、`duration`、`protection`；减速、冻结 | 脆裂、冰雾、阻挡 | 燃烧、连锁电击、强推 |
| 雷 | `speed`、`propagation`、`penetration`；感电、连锁 | 过载、震裂、瞬发 | 稳定推拉、实体墙、长时间冻结 |
| 岩 | `force`、`protection`、`control`；震晕、破甲、阻挡 | 熔裂、带电震击、扬尘 | 天然导电、连锁、急速位移 |
| 影 | `penetration`、`duration`、`control`；标记、束缚、延迟 | 余烬、冰蚀、雷影回响 | 直接护墙、强推拉、纯治疗 |
| 风 | `speed`、`area`、`force`、`propagation`；推、拉、加速 | 助燃、冰雾、雷暴扩散 | 冻结、实体护墙、岩石破甲 |

### 5.1 “禁止”的工程含义

禁止不是把整条元素关系置零，而是：

- 对不合理机制的 `capabilityMultiplier` 设为 `0`。
- 其他合理参数仍可变化。
- Trace 必须记录该贡献因能力边界被裁掉，便于 How 解释。

例如雷作用于岩：

- 可以增加瞬时冲击、震裂或爆发。
- 可以削弱岩构造的稳定性。
- 不应自动给普通岩石增加“导电”标签。

## 7. Seed Aspect 的保留方式

为兼容现有 ID，继续保留：

```ts
type SeedAspect = "impact" | "flow" | "zone" | "mark";
```

但它只表示设计原型，不再等同于最终形态：

| Aspect | 主要职责 | 常见参数倾向 |
|---|---|---|
| `impact` | 单次命中、爆发、冲击 | 威力、速度、力量、穿透 |
| `flow` | 持续输出、移动、传播 | 持续、速度、传播 |
| `zone` | 空间覆盖、控制、守护 | 范围、持续、控制、防护 |
| `mark` | 延迟、构造、触发、召唤 | 持续、控制、延迟爆发 |

因此 `frost_mark` 可以是冰墙，`wind_mark` 可以是飓风之眼。编码时不要通过字符串后缀推断 `SkillForm`，必须读取基础技能数据。

## 8. 数据结构合同

```ts
type NormalizedStats = Record<StatKey, number>;
type MechanicState = Partial<Record<MechanicKey, number>>;

type BaseSkillDefinition = {
  id: string;
  name: string;
  element: ElementKey;
  aspect: SeedAspect;
  form: SkillForm;
  coreEffect: string;
  baseStats: NormalizedStats;
  nativeMechanics: MechanicState;
  visualProfile: {
    trajectory?: AnimationTrajectory;
  };
};
```

强约束：

- `baseStats` 必须完整包含 10 个键。
- `nativeMechanics` 可以稀疏。
- 所有数值在模块加载时校验为 `0..1`。
- `form`、`element`、`coreEffect` 在生成结果中保持可追溯。
- 任何互馈新增机制必须通过元素能力边界校验。

## 9. 调参边界

24 个基础值是 V6 第一版默认值，不是永久平衡值。允许调参，但需满足：

1. 不改变技能主效果和固定形态。
2. 单次基础技能仍能独立看懂。
3. 同元素四技能的参数轮廓明显不同。
4. 不为制造 A/B 差异而单独修改默认案例。
5. 每次调参后运行全量顺序性质测试。
