# 非对称双向互馈引擎合同

## 0. 文档定位与优先级

本文档记录 2026-06-12 经过人工观察、测试与讨论后确定的下一代引擎方向。

它不是当前代码已经完成的功能说明，而是后续修改 `src/engine/`、Play、How 和 Three.js 动画时必须遵守的设计合同。

具体的数据表、计算公式、输出类型、展示和迁移步骤已经细化到 `docs/v6/README.md` 所索引的 V6 可执行设计包。编码时以 V6 设计包为直接实现依据，本文档用于理解设计背景与不可违背的结论。

当本文档与以下旧文档发生冲突时，以本文档为准：

- `docs/contracts/data-structures.md`
- `docs/contracts/engine-pipeline.md`
- `AI规则引擎Demo_核心规则引擎修正文档_V4_向量互馈解码版.md`
- `AI规则引擎Demo_V5_展示实时互馈优化文档.md`

仍然必须遵守 `docs/CONSTITUTION.md` 和 `AGENTS.md` 中的最高优先级规则：

1. 纯前端本地规则引擎。
2. 禁止完整排列查表。
3. 核心规则写成 `src/engine/` 下的纯函数。
4. 每次生成必须输出真实 `trace`。
5. 同一组种子的不同顺序必须产生明显差异。

---

## 1. 本轮要解决的问题

当前 V4/V5 实现虽然存在向量、互馈、候选能力和动画参数，但人工体验中出现了以下问题：

1. 元素交互缺少可信依据，例如岩与雷被泛化为“导电”，不符合普通岩石的直觉。
2. 许多元素组合只有正向加成，没有抑制、转化或弱关系，导致技能逐渐拥有相似的效果集合。
3. 前向和后向影响传递的是近似相同的行为数据，没有承担不同的设计职责。
4. 互馈经过多层缩放后过弱，Play 和 How 中难以观察到真实变化。
5. 单个技能同时选择过多能力，主效果不清晰，动画也无法表达重点。
6. How 页展示了大量结果数值，却没有清楚回答“谁改变了谁、改变了什么、为什么选择这个变化”。

本轮目标不是增加更多计算，而是让每一次有效互馈都满足：

```text
合理
+ 有方向
+ 有取舍
+ 可观察
+ 可解释
```

## 2. 核心设计结论

### 2.1 四个槽位仍是四个独立技能

4 个槽位不是同一个技能的四个动作阶段，而是 4 个独立技能位。

```text
技能 I -> 技能 II -> 技能 III -> 技能 IV
```

排列顺序代表技能的固化顺序和互馈拓扑。

### 2.2 一个技能必须先有清晰的基础主效果

每枚技能种子首先定义一个可独立理解的基础技能，不再让一枚种子天然携带多个同等重要的效果。

建议结构：

```ts
type BaseSkillDefinition = {
  seedId: string;
  element: ElementKey;
  baseShape: SkillShape;
  baseEffect: CoreEffect;
  baseParams: SkillParams;
  semantic: SkillSemanticState;
};
```

例如：

| 种子 | 基础主效果 |
|---|---|
| 爆焰 | 远距离火焰弹，命中后爆炸 |
| 灼流 | 持续向前喷射或流动的火焰 |
| 炎域 | 在地面生成持续燃烧区域 |
| 烬印 | 标记目标并延迟引爆 |

最终技能建议限制为：

```text
基础主效果
+ 最多 1 个显著前向本质变异
+ 1 个由全部后位来源联合计算得到的后向修饰包
```

不得再默认生成“1 个主能力 + 2 个辅助能力 + 1 个变异能力”的效果堆叠。

### 2.3 双向互馈必须非对称

系统保留“双向注意力”的概念，但两个方向承担不同职责。

```text
Forward Attention
前位 -> 后位
职责：改变后续技能的元素表现、核心机制或主要参数

Backward Attention
后位 -> 前位
职责：为前面技能增加轻量附加效果、触发方式或参数修正
```

前向影响强而局部，后向影响轻但具有“最终选择定调”的权威性。

## 3. 语义维度设计

### 3.1 维度是公共语言，不是元素效果清单

所有元素可以在同一语义空间中计算，但不代表所有元素都能自然使用全部维度。

错误理解：

```text
系统存在 pushPull 维度
=> 雷技能也可以自然产生推拉
```

正确理解：

```text
系统存在 pushPull 维度
=> 引擎可以表达推拉
=> 纯雷是否能获得推拉，由元素能力边界和反应解锁规则决定
```

### 3.2 建议将语义拆成三层

不建议继续把形态、节奏、功能和结果全部放进同一个平面向量竞争。

#### 元素本质

保留 6 维：

```text
fire, frost, lightning, stone, shadow, wind
```

#### 主形态

建议作为离散类型或较小的形态向量：

```text
projectile  投射
beam        射线/持续指向
zone        领域
chain       连锁路径
movement    位移技
summon      召出物/结构物
```

主形态来自基础技能，并在整个互馈过程中保持只读。

投射仍然是投射，领域仍然是领域，结构物仍然是结构物。互馈只能改变：

- 元素表现。
- 核心机制。
- 命中或持续过程。
- 参数。
- 次级视觉信号。

例如火焰投射物受到冰影响后，可以变为带减速和冰晶碎裂的冰火投射物，但不能直接变成冰霜领域。

#### 行为效果

首轮建议收缩为 10-12 个：

```text
impact      冲击
burst       爆发
duration    持续
spread      扩散
control     控制
bind        禁锢
mark        标记
pierce      穿透
guard       防护
pushPull    推拉
mobility    位移
propagation 传播
```

是否保留 `speed`、`delay`、`summon` 为独立连续维度，属于待实现审计项：

- `speed` 可以成为基础参数，不一定需要竞争。
- `delay` 可以成为触发方式，不一定需要和伤害效果同层。
- `summon` 更接近主形态，建议移出行为效果层。

### 3.3 技能数量不需要覆盖所有维度组合

当前 24 枚种子（6 元素 x 4 个基础原型）足以支持 MVP。

技能数量不应随着维度组合指数增长。变化来自：

```text
基础技能
+ 合法的前向本质变异
+ 合法的后向轻量附加
```

而不是为每一种维度组合预写完整技能。

## 4. 元素能力边界

### 4.1 每个元素必须声明能力边界

建议数据结构：

```ts
type ElementCapabilityProfile = {
  native: BehaviorKey[];
  possible: BehaviorKey[];
  reactionOnly: BehaviorKey[];
  forbidden: BehaviorKey[];
};
```

- `native`：可以自然成为该元素技能的主效果。
- `possible`：允许出现，但通常不是最强倾向。
- `reactionOnly`：只有特定跨元素反应可以解锁。
- `forbidden`：当前世界规则下不允许出现。

### 4.2 首轮能力边界建议

此表是首轮实现基线，后续可以根据世界观调整，但不能让所有元素趋同。

| 元素 | native | possible | reactionOnly / 单元素极弱 | forbidden |
|---|---|---|---|---|
| 火 | 爆发、持续、扩散 | 冲击、标记 | 禁锢、防护、位移 | 暂无永久禁止项 |
| 冰 | 控制、禁锢、领域留存 | 穿透、防护、持续 | 连锁、推拉、高速位移 | 暂无永久禁止项 |
| 雷 | 冲击、传播、高速释放 | 爆发、标记、穿透 | 推拉、防护、持续领域 | 暂无永久禁止项 |
| 岩 | 防护、禁锢、穿透、结构 | 冲击、领域 | 连锁、高速、位移 | 暂无永久禁止项 |
| 影 | 标记、延迟触发、穿透 | 持续、位移、爆发 | 正面防护、推拉 | 暂无永久禁止项 |
| 风 | 推拉、位移、传播 | 扩散、穿透、防护 | 禁锢、持续伤害、标记 | 暂无永久禁止项 |

这里的“高速释放”和“延迟触发”可以最终落为参数或触发类型，不一定保留为行为维度。

MVP 优先使用 `reactionOnly` 表达“单元素不能自然产生，但特定反应可以解锁”。`forbidden` 只为未来明确的世界规则保留，不应过早把扩展空间锁死。

### 4.3 跨元素反应用于解锁，不用于绕过边界

例如：

```text
纯雷：
传播 = 合法
推拉 = 不合法

风 -> 雷：
解锁推拉或位移修饰
原因：风场改变雷电传播路径，形成雷暴推进
```

```text
纯火：
禁锢 = 不合法

岩 -> 火：
允许局部禁锢
原因：熔岩冷却形成岩壳，而不是“火焰天然会禁锢”
```

所有反应必须有可读的机制解释。

## 5. 元素交互规则

### 5.1 元素关系必须有方向

`fire -> frost` 与 `frost -> fire` 不应共享完全相同的结果。

建议结构：

```ts
type DirectedElementReaction = {
  source: ElementKey;
  target: ElementKey;
  strength: number;
  kind: 'amplify' | 'transform' | 'suppress' | 'anchor' | 'weak';
  unlocks?: BehaviorKey[];
  boosts?: Partial<Record<BehaviorKey, number>>;
  suppresses?: Partial<Record<BehaviorKey, number>>;
  elementTransfer?: number;
  shapeMutation?: SkillShape;
  explanation: string;
};
```

该表描述的是两种元素之间的基础规律，不是四槽完整排列查表，因此不违反项目约束。

### 5.2 关系不必全部是正向增益

元素关系至少允许：

- `amplify`：风助火势，强化已有机制。
- `transform`：火冰形成蒸汽、融化或冻裂。
- `suppress`：岩使雷失去部分传播，冰削弱火焰持续。
- `anchor`：岩把风的位移倾向锚定为风墙或局部旋流。
- `weak`：没有可信强反应时，只产生很小影响或不产生显著变异。

### 5.3 岩与雷的基线解释

普通岩石不应默认解释为导电材料。

首轮建议：

```text
雷 -> 岩：
雷击造成热冲击和震裂
增强：冲击、爆发、穿透
削弱：传播、连锁

岩 -> 雷：
岩层使雷电接地、集中或被阻断
增强：单点冲击、延迟地面震荡
削弱：传播范围、连锁数量
```

只有未来明确引入“金属矿脉”“导电晶体”等种子时，才允许把岩雷解释为导电连锁。

### 5.4 火与冰的基线解释

```text
火 -> 冰：
火的本质进入后续冰技能
候选变化：融化、蒸汽扩散、冷热裂变
可能牺牲：禁锢强度、冻结时长
可能获得：扩散、爆发、持续灼伤

冰 -> 火：
冰作为后续选择反向修饰前面的火
火技能本体仍然保留
可能附加：结霜、减速、冰晶碎裂
可能牺牲：少量爆发或持续强度
```

## 6. 前向注意力

### 6.1 设计职责

前位技能对后位技能的影响接近“本质改变”。

它可以传递：

- 元素残留。
- 核心行为。
- 定向元素反应。

它可以产生：

- 元素混合。
- 核心机制替换。
- 明确的获得与牺牲。

前向影响不能修改基础主形态。

### 6.2 距离越近，影响越强

首轮建议距离权重：

```ts
const FORWARD_DISTANCE_WEIGHT = {
  1: 1.0,
  2: 0.45,
  3: 0.20,
};
```

例如在：

```text
火 -> 冰 -> 雷 -> 风
```

风主要受到雷的塑形，冰保留次级影响，火只留下轻微底色。

### 6.3 所有前向来源连续参与，不使用强制归一化的普通 Softmax

普通 Softmax 会强行把总权重分配给所有来源，即使所有关系都很弱，也会制造反应。

确定流程：

```text
独立计算每个前向来源对各效果维度的有符号贡献
-> 同方向贡献以递减收益方式增强
-> 反方向贡献相互抵消
-> 使用连续饱和函数限制最大变化
-> 将联合结果翻译为一个主要前向变异包
```

硬性约束：

- 不按 Top-K 丢弃计算来源。
- 不使用效果生效阈值切断弱关系。
- 弱关系因兼容度和距离衰减自然接近零。
- 对外展示只突出联合结果中最显著的 1-3 项变化。
- 如果所有变化都极弱，技能保持接近基础状态，不强造混合效果。

### 6.4 前向计算建议

```text
forwardScore(source -> target)
= 语义适配度
 x 定向元素反应强度
 x 距离权重
 x 目标技能对该变异的可接受度
```

前向 Value 不应直接复用完整行为向量，而应包含：

```text
元素本质
+ 核心行为
+ 反应解锁项
+ 反应抑制项
```

### 6.5 混合元素不继续向后传播

技能受到前向影响后，可以在自身最终结果中出现一个次级元素，但该次级元素只属于当前技能。

例如：

```text
火 -> 冰 -> 雷

冰技能可以成为“冰 + 火”的混合技能
冰技能继续影响雷技能时，仍只按“冰”作为来源元素
火元素不会经由冰继续传播到雷
```

每个技能向后输出的来源签名为：

```text
基础主元素
+ 该主元素仍然保留的原生行为强度
```

外来次级元素、外来解锁机制和后向附加效果均不进入后续传播。

这样既允许技能自身发生明显混合，又防止四槽结束时所有技能都变成多元素杂烩。

## 7. 后向注意力

### 7.1 设计职责

后位技能对前位技能主要产生轻量修饰，不应轻易替换前位技能本体。

它可以传递：

- 附加状态。
- 命中后的二段效果。
- 触发条件。
- 小幅参数变化。
- 统一 Build 风格的收束特征。

例如：

```text
火球 <- 后位冰技能

火球仍然是火球
+ 命中后短暂结霜
+ 爆炸边缘产生冰晶碎片
```

### 7.2 越靠后的来源，反向权威越高

后位技能的槽位越靠后，越代表玩家的最终选择，对前面技能的定调能力越强。

首轮建议来源权威：

```ts
const BACKWARD_SOURCE_AUTHORITY = {
  slot2: 0.35,
  slot3: 0.65,
  slot4: 1.00,
};
```

允许再叠加较弱的目标距离衰减：

```ts
const BACKWARD_DISTANCE_WEIGHT = {
  1: 1.00,
  2: 0.85,
  3: 0.70,
};
```

这里必须保证“槽位越靠后，来源权威越高”是主导因素，不能被距离衰减完全抵消。

### 7.3 所有后向来源联合计算

```text
backwardScore(source -> target)
= 附加效果适配度
 x 来源槽位权威
 x 目标距离权重
 x 元素反应合法性
```

后向 Value 只允许包含：

```text
附加效果
+ 触发方式
+ 参数修正
+ 轻量视觉修饰
```

默认禁止后向 Value 直接替换目标的基础主形态和主元素。

所有后位技能都参与目标技能的后向计算，不预先限制“最多影响几个目标”，也不按门控阈值删除来源。

例如技能 I 的后向来源为技能 II、III、IV：

```text
技能 II 提供减速
技能 III 相性很弱，贡献接近零
技能 IV 提供扩散和速度

联合结算后：
技能 IV 可以削弱技能 II 原有修饰的一部分
也可以与技能 II 在不同维度并存或形成协同
```

建议按效果维度进行有符号聚合：

```text
contribution
= reactionDelta
 x mechanismCompatibility
 x sourceAuthority
 x distanceWeight

resolvedDelta
= saturate(sum(contribution))
```

其中 `saturate` 使用 `tanh` 或指数饱和等连续函数：

- 同方向来源可以增强，但收益递减。
- 反方向来源自然抵消。
- 不会因为增加多个微弱来源而无限叠加。
- 不需要 Softmax，也不会把负向影响错误转成正权重。

联合结算可以得到一个包含多项变化的“后向修饰包”，但对外最多突出 1 个机制变化和 2 个主要参数变化。

### 7.4 最终槽位的意义

技能 IV 是构筑的“最终落子”：

- 它仍然是一个独立技能。
- 它对前面技能的反向修饰权威最高。
- 它可以让整套 Build 呈现统一的收束风格。
- 它不能把前面三个技能全部改成自己的复制品。

### 7.5 后向修饰不递归传播

每次构筑变化都从基础技能全量重算，但单次计算内部必须保持无循环：

```text
1. 恢复基础技能
2. 计算并冻结全部前向结果
3. 基于冻结后的前向结果生成后向贡献
4. 对每个目标联合结算后向贡献
5. 输出最终技能
```

技能 IV 修改技能 II 后，修改后的技能 II 不会再次返回去重新修改技能 I。

UI 可以展示连续变化，但引擎不能进行递归迭代，否则结果会依赖计算次数并失去可解释性。

## 8. 技能结果结构

建议将最终技能显式拆为基础、前向变异和后向附加：

```ts
type GeneratedSkillV6 = {
  slotIndex: number;
  seedId: string;

  base: {
    name: string;
    element: ElementKey;
    shape: SkillShape;
    effect: CoreEffect;
    params: SkillParams;
  };

  forwardMutation?: {
    sourceSlots: number[];
    reactionIds: string[];
    name: string;
    transferredElement?: ElementKey;
    gained: EffectDelta[];
    lost: EffectDelta[];
    explanation: string;
  };

  backwardModifier?: {
    sourceSlots: number[];
    modifierIds: string[];
    gained: EffectDelta[];
    lost: EffectDelta[];
    explanation: string;
  };

  final: {
    name: string;
    elements: ElementKey[];
    shape: SkillShape;
    params: SkillParams;
    tags: string[];
    animationParams: AnimationParams;
  };
};
```

最终结果必须能直接回答：

```text
这个技能原本是什么？
前面的哪个技能改变了它的本质？
后面的哪个技能给它追加了什么？
它获得了什么？
它牺牲了什么？
动画如何表达这次变化？
```

## 9. Trace 合同

How 页不得只展示最终数值。Trace 必须记录完整因果链。

建议结构：

```ts
type SkillMutationTrace = {
  slotIndex: number;
  seedId: string;
  baseline: {
    element: ElementKey;
    shape: SkillShape;
    params: SkillParams;
    topBehaviors: BehaviorScore[];
  };

  forwardCandidates: Array<{
    sourceSlot: number;
    reactionId?: string;
    score: number;
    contribution: ParamDelta[];
    reason: string;
  }>;

  selectedForwardMutation?: {
    sourceSlots: number[];
    reason: string;
    deltas: ParamDelta[];
  };

  backwardCandidates: Array<{
    sourceSlot: number;
    authority: number;
    score: number;
    contribution: ParamDelta[];
    reason: string;
  }>;

  selectedBackwardModifier?: {
    sourceSlots: number[];
    reason: string;
    deltas: ParamDelta[];
  };

  final: {
    elements: ElementKey[];
    shape: SkillShape;
    params: SkillParams;
    topBehaviors: BehaviorScore[];
  };
};
```

How 页优先展示自然语言因果：

```text
冰域位于爆焰之后
-> 对爆焰提出“结霜碎裂”附加
-> 该附加通过元素合法性检查
-> 爆发 75 -> 68
-> 控制 8 -> 34
-> 动画增加命中后的冰晶环
```

原始小数分数可以放在详情中，但不能成为主要解释。

## 10. Play 页变化表达

每次添加、移除或重排种子后，Play 页必须明确显示：

1. 哪个技能发生了变化。
2. 变化由哪个槽位触发。
3. 基础主效果是否改变。
4. 获得了什么。
5. 牺牲了什么。
6. 参数从多少变成多少。
7. 动画新增了哪个可辨识信号。
8. 本轮新增种子让上一轮结果发生了哪些变化。

建议展示格式：

```text
基础：远程火焰爆炸
前向变异：蒸汽冻裂（来自技能 I）
后向附加：命中后风压击退（来自技能 IV）

爆发 75 -> 62
控制 8 -> 41
范围 42 -> 58
```

不允许仅通过技能名变化或几条相近的参数条让用户自行猜测。

Play 页每次都对比前后两个完整 Build 快照，并至少展示：

```text
新增影响来源
消失的影响来源
参数 Delta
机制新增或移除
最终动画信号变化
```

## 11. 动画表达合同

动画必须采用：

```text
一个主形态
+ 一个前向本质变异信号
+ 最多一个后向轻量信号
```

例如：

```text
主形态：远程火焰弹体爆炸
前向冰变异：飞行过程出现蒸汽，命中后形成冰晶环
后向风附加：爆炸后产生短促风压
```

禁止同时无重点地播放：

- 连锁。
- 禁锢环。
- 延迟标记。
- 地面领域。
- 爆裂。
- 击退。

除非 Trace 明确证明这些机制被选中。

动画区域应标注当前正在表达的语义：

```text
主形态：投射爆炸
本质变异：冰火冻裂
附加效果：风压击退
```

视觉差异必须同时来自最终技能结果，不得在 Three.js 组件中按完整排列硬编码。

## 12. 默认案例要求

继续保留：

```text
A: fire -> frost -> lightning -> wind
B: wind -> lightning -> frost -> fire
```

实际使用时应选择四个明确的技能种子，而不是只有元素名称。

A/B 必须满足：

1. 主形态保持各自基础定义不变。
2. 至少 2 个技能的核心机制或主要参数明显不同。
3. 至少 2 个技能具有不同的前向本质变异。
4. 技能 IV 对前面技能产生可观察的收束修饰，其实际强度由联合计算决定。
5. Play 页能明确列出变化来源和参数差值。
6. How 页能从 Trace 解释每个差异。
7. 动画的变异信号肉眼可辨。
8. 不依赖技能名称变化伪装差异。
9. 混合元素不跨技能继续传播。
10. 正序与倒序至少形成两种可用一句话区分的 Build 风格。

### 12.1 自动差异验收

引擎测试至少覆盖：

1. 所有有向元素对都能稳定、确定地生成影响结果。
2. `A -> B` 与 `B -> A` 的影响维度、方向或机制至少有一项不同；若刻意对称，必须在规则中写明原因。
3. 每次添加第 2、3、4 个默认案例种子后，至少有一个已有技能产生可展示变化。
4. 默认正序与倒序不能只改变名称，必须改变机制、参数或动画信号。
5. 混合元素只存在于当前技能，不会出现在后续技能的来源签名中。
6. 同一输入重复计算结果完全一致。
7. 所有参数保持在合法范围内，不产生 `NaN`、无穷值或负持续时间。

## 13. 实现顺序建议

后续 coding agent 不应直接一次性改完整 UI。建议按以下顺序实施：

1. 审计现有 24 枚种子的基础主效果，删除效果堆叠。
2. 定义分层语义结构和元素能力边界。
3. 定义有方向的元素反应表。
4. 实现前向本质变异通道。
5. 实现后向轻量附加通道。
6. 输出新的因果 Trace 和参数 Delta。
7. 用纯函数测试验证差异、元素守恒和确定性。
8. 更新 Play 页变化表达。
9. 更新 How 页因果解释。
10. 最后更新 Three.js 动画映射。

每一步都必须先证明引擎输出正确，再修改展示层。

## 14. 禁止事项

- 禁止为四槽完整排列编写结果查表。
- 禁止让所有元素都自然获得全部行为效果。
- 禁止为了“有交互”而给每对元素强行添加正向共鸣。
- 禁止普通岩石默认产生导电连锁。
- 禁止使用 Softmax 强迫弱关系分配影响。
- 禁止混合元素、外来机制或后向附加继续向后传播。
- 禁止一个技能默认携带多个同等重要的主能力。
- 禁止后向影响轻易替换前位技能本体。
- 禁止后向修改结果在同一轮计算中递归回流。
- 禁止只修改技能名称而参数、机制和动画基本不变。
- 禁止 How 页用手写文案解释一个 Trace 中不存在的变化。
- 禁止动画播放未被最终技能结果选中的效果。

## 15. 已确认实现决策与校准项

### 15.1 元素反应口径

MVP 以普通玩家容易理解的常见直觉为主：

- 蒸汽。
- 融化。
- 冻裂。
- 风助火势。
- 风雷扩散。
- 雷击震裂。
- 岩层接地或阻断。

玄幻概念只用于命名和包装，不作为理解规则的前置条件。

### 15.2 主形态固定

主形态为基础技能只读字段，互馈不得修改。

机制和视觉可以发生显著变化，但必须保留基础动作轮廓。

### 15.3 技能 IV 与所有后向来源

技能 IV 权威最高，但没有“必定覆盖前三个技能”的特殊规则。

所有后位来源都参与连续联合计算：

- 相性高则产生明显贡献。
- 相性低则自然接近零。
- 同方向增强。
- 反方向抵消。
- 不同维度可以并存。

### 15.4 计算阈值与展示策略

计算层不使用效果生效硬阈值。

UI 不使用单一百分比阈值决定“展示或不展示”，而是：

1. 永远展示机制新增、机制移除和元素混合。
2. 将参数变化按绝对幅度排序。
3. 默认展示最大的 1-3 项参数变化。
4. 其余来源折叠到详细 Trace。
5. 只使用极小 epsilon 清理浮点噪声，不参与设计判定。

为了保证 Demo 可观察，默认 A/B 案例必须通过自动测试证明：

- 至少 2 个技能存在机制差异，或
- 至少 3 个关键参数存在足够大的视觉或数值差异。

### 15.5 候选能力库的职责

当前候选能力库可以保留概念，但建议改变职责：

- 基础技能不再通过候选竞争决定。
- 前向变异和后向附加可以分别拥有小型候选库。
- 候选库用于翻译计算结果，不用于给技能堆叠多个完整能力。

### 15.6 注意力实现方式

设计目标是获得类似双向注意力的关系计算能力，不要求机械复刻 Transformer。

首轮确定使用：

```text
方向独立的关系评分
+ 有向元素反应
+ 距离或槽位权威
+ 分维度有符号聚合
+ 连续饱和函数
```

如果未来引入 Q/K/V，也必须继续遵守：

- 前向和后向使用不同的 Value 内容。
- 弱关系可以自然趋近零。
- 不使用 Softmax 强迫所有来源参与。
- 输出必须保持确定、可追踪、可解释。

### 15.7 有向元素反应表的覆盖范围

6 种元素共有 30 个不同元素的有向关系，另有 6 个同元素关系。

为了支持任意种子组合，首轮必须为全部 36 个方向提供基础反应轮廓：

- 增强。
- 削弱。
- 转化。
- 锚定。
- 弱关系。
- 同源共鸣。

基础轮廓只需要给出直觉解释和少量参数方向，不需要每一对都有独特复杂机制。

复杂的命名反应优先定义：

1. 默认 A/B 案例涉及的关系。
2. 玩家最容易产生直觉预期的关系，例如火冰、风火、风雷、冰雷、岩火、岩雷。
3. 动画能够低成本清楚表达的关系。

其余关系可以使用简单的 `weak`、增强或抑制轮廓，但不能缺失规则后回退为随意正向加成。

每新增一条反应都必须回答：

```text
为什么会发生？
方向反过来是否不同？
获得了什么？
牺牲了什么？
动画如何表达？
```

### 15.8 二十四个基础技能的工程化定义

专业设计回复提供的 24 枚技能作为内容草案使用，但不能原样照搬复杂战斗机制。

在实现新互馈前，应先完成一张基础技能表，确保：

- 每枚种子脱离构筑时也能独立理解。
- 同一元素的 4 枚种子主形态或核心机制有明确差异。
- 基础技能不依赖候选能力竞争才知道自己是什么。
- 动画可以先完整表达基础技能，再叠加互馈信号。
- 每枚基础技能只保留 1 个核心效果、1 个原生状态和 4-6 个可视参数。
- 不引入当前 Demo 无法验证的仇恨、友军、暴击流派、复杂战斗条件等系统。
